# ADR-0004: 웹 리서치 쉘과 feature-first 스캐폴드 채택

- 상태: 승인
- 날짜: 2026-03-16

## 배경
`apps/web` 는 기본 Next.js 템플릿 상태였고, 메인 4개 화면을 금융 리서치 워크스페이스로 확장할 공통 shell 과 데이터 구조가 없었다.

## 결정
웹앱 구조를 아래 원칙으로 스캐폴딩한다.

- `app/*/page.tsx` 는 thin wrapper 로 유지한다.
- 공통 shell 은 sidebar + topbar + theme toggle 구조로 통일한다.
- provider 는 theme 와 query 를 루트에서 묶는다.
- UI 조합은 `features/*` 에, 개발용 데이터는 `dev/fixtures/*` 에 둔다.
- grid 는 AG Grid Community 공통 래퍼와 한국어 locale 을 사용한다.

## 이유
- route 레이어를 얇게 두면 화면 로직이 `page.tsx` 에 몰리지 않는다.
- feature-first 구조가 메인 4개 화면과 공통 기능(grid/chart/filter/watchlist/sector/history)을 같이 확장하기 쉽다.
- fixture 기반 스캐폴드가 API 미연동 단계에서도 UI 검증과 문서 정합성을 유지하게 한다.

## 영향
- `apps/web/app`, `apps/web/components`, `apps/web/features`, `apps/web/dev/fixtures` 구조가 공통 기준이 된다.
- 다크모드, AG Grid locale, 숫자 토큰, 카드 간격 토큰이 공통 디자인 기준으로 고정된다.
- 이후 API 연동은 fixture loader 를 실제 서버 상태 로더로 대체하는 방식으로 진행한다.

## 단점 / 트레이드오프
- 초기 단계에서는 fixture 와 실제 API 계약이 이중으로 존재한다.
- 공통 shell 과 토큰을 먼저 고정했기 때문에 세부 화면은 이후 조정이 필요할 수 있다.

## 후속 작업
- 실제 API 연결 시 page loader 를 서버 fetch 로 교체한다.
- overview/radar/stock/history 각 화면을 실제 데이터 모델과 테스트로 확장한다.
