# ADR-0010: 브랜치 승격 정책

- Status: Approved
- Date: 2026-04-19

## Background

`stock`는 `develop_loop`, `develop`, `main`을 장수 브랜치로 사용한다.

기존에는 작은 작업 PR과 장수 브랜치 승격을 모두 같은 방식으로 다루면서 다음 문제가 반복됐다.

- `issue/* -> develop_loop`에는 `squash`가 적합하지만
- `develop_loop -> develop`, `develop -> main`까지 `squash`로 처리하면
- 실제 내용은 같아도 조상 관계가 끊겨 다음 승격 PR이 `dirty` 상태가 된다
- 그 결과 `develop -> main`만 허용하는 릴리스 규칙과 실제 Git 이력이 충돌한다

## Decision

- `issue/* -> develop_loop`는 계속 `squash`를 사용한다.
- `develop_loop -> develop`는 `merge`를 사용한다.
- `develop -> main`은 반드시 `merge`를 사용한다.
- `main` 대상 PR은 항상 `develop -> main`만 허용한다.
- `main` 대상 PR 작성자는 `alpaka206`만 허용한다.
- `main`, `develop`, `develop_loop`에는 직접 커밋과 직접 push를 금지한다.
- 장수 브랜치 승격을 막는 `required_linear_history`는 `develop`, `main`에서 강제하지 않는다.

## Why

- 작은 작업 단위는 `squash`가 히스토리를 깔끔하게 정리한다.
- 장수 브랜치 승격은 조상 관계 보존이 더 중요하다.
- `develop -> main`만 허용하는 정책을 실제 운영에서도 지키려면 `merge` 기반 승격이 필요하다.
- 보호 규칙, 자동화 규칙, 실제 Git 이력이 같은 방향을 보아야 릴리스 자동화가 안정적으로 동작한다.

## Impact

- 릴리스 흐름은 `issue/* -> develop_loop -> develop -> main`으로 유지된다.
- `develop_loop -> develop`, `develop -> main` PR은 merge commit이 허용되어야 한다.
- `release-guard`는 계속 `develop -> main`만 통과시킨다.
- Ralph/Codex 설정 파일과 문서는 이 승격 정책을 source of truth로 삼는다.
